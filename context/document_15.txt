DramaRama — Complete Application Restructuring
From visible 5 Elements workspace to invisible scaffolding chatbot with Deep Understanding
March 2026


========================================
PART 1: WHAT DRAMARAMA BECOMES
========================================

DramaRama is a platform that helps anyone develop deeper understanding of the problems they face. The user describes a real problem. A smart chatbot — powered invisibly by the 5 Elements of Effective Thinking — asks questions that drive the user to think deeper. A Deep Understanding Document builds in real time on the left side of the screen as the conversation progresses. The user walks away understanding their problem better than when they arrived.

The 5 Elements are the engine. The user never sees them. They experience the result: deeper understanding.

The website communicates nothing about software engineering specifically. It is domain-agnostic. Anyone can use it. Marketing (outside the website) will target software engineers first, but the product itself serves anyone.


========================================
PART 2: DESIGN LANGUAGE
========================================

2.1 Overall Aesthetic

The design language is: MATTE PROFESSIONAL FUTURISM.

- Matte, not shiny. Professional, not gaming. If matte were something you could touch and smell, that is the feeling.
- Dark backgrounds (near-black, deep charcoal) with matte surface textures.
- Accent colors are muted and purposeful — not neon, not gradient-heavy.
- Typography is clean, modern, with generous spacing. No playful fonts.
- Everything hints at depth and intelligence without being flashy.
- The feeling when you land on the site: "This is serious. This is sophisticated. This is something real."

2.2 Color System

Primary background: #0D0D0D (near black) to #1A1A1A (dark charcoal)
Secondary background: #242424 (panels, cards)
Text primary: #F5F5F5 (off-white)
Text secondary: #888888 (muted gray)
Text accent: #C0C0C0 (silver)

Element colors (used subtly in the Deep Understanding Document and Framework page — NOT as dominant UI colors):
- Earth: #8B6F47 (warm brown, matte)
- Fire: #B84233 (deep muted red)
- Air: #7A8B99 (steel blue-gray)
- Water: #3D6B7E (deep teal)
- Change: #9B8EC4 (muted purple)

Accent highlight: #E8E0D4 (warm off-white, for important text and borders)

No bright colors. No neon. No gradients. Matte and purposeful.

2.3 Typography

Headings: A clean sans-serif (Inter, or similar). Weight 600-700. Generous letter-spacing.
Body text: Same family, weight 400. Line height 1.7 for readability.
Monospace (for document content): JetBrains Mono or similar.
No decorative fonts. No serif. Clean and modern.

2.4 Components

Cards/panels: Subtle border (1px #333333), no shadow or very subtle shadow. Slightly lighter background than the page (#242424 on #0D0D0D).
Buttons: Matte, solid fill, no gradient. Primary button: off-white text on dark background with border. Hover: subtle brightness increase.
Inputs/textareas: Dark background (#1A1A1A), light text, thin border. No rounded corners or very subtle (2-4px).
Spacing: Generous. Let the content breathe. Nothing cramped.

PART 3: THE THREE PIECES
========================================

3.1 PIECE 1 — THE WORKSPACE (Main Experience)

This is the core of the app. The user comes here to develop deeper understanding.

Layout: Two-column on desktop. Full width on mobile.

LEFT COLUMN (40% width): Deep Understanding Document
- Header: "YOUR UNDERSTANDING" in small caps, tracking-widest, muted text
- Below: modular blocks that accumulate as the conversation progresses
- Each block is a rectangular element with:
  - A thin left border in the element color (but NO element name visible to the user)
  - The insight text in clean monospace font
  - Blocks stack vertically and connect visually (a thin vertical line connecting them, like a timeline or neural pathway)
- When a new insight is extracted, it fades in (Framer Motion) and the connecting line extends
- When empty: "Your understanding will build here as you explore your problem..."
- This column scrolls independently

RIGHT COLUMN (60% width): The Chatbot
- Top: The user's problem description displayed in a compact card with the label "YOUR PROBLEM" in small caps
- Below: A conversation thread — user messages and chatbot messages in a clean chat layout
  - User messages: right-aligned, dark card with light text
  - Chatbot messages: left-aligned, slightly lighter card, with a subtle accent border
  - No emojis, no "COACH" label, no element names visible
- Bottom (sticky): A text input area with a "Send" button
  - Placeholder: "What are you thinking about this problem?"
  - The button text is just "Send" or a send icon — NOT "Get Nudge" or "Build Understanding"
- The chatbot should feel like a conversation with a very smart colleague, not a coaching interface

How the chatbot works (invisible to user):
- When the user sends a message, the backend determines which element would best serve the user's current thinking based on what they wrote
- The LLM uses the 5 Elements invisibly to craft a response that drives deeper understanding
- After the chatbot responds, a background call extracts the key understanding and adds it to the left column
- The user never sees element names, prompt names, or anything about the framework
- The conversation is a single thread — NOT 13 separate element chatbots

This is a MAJOR change from the current architecture: instead of 13 separate prompts with separate conversation histories, there is ONE conversation thread. The backend invisibly decides which element to apply based on the user's message.

Session flow:
1. User arrives at workspace
2. If no active session: show a centered input asking "What problem are you facing?" with a large textarea and a "Start" button
3. User describes their problem and hits Start
4. Two-column layout appears: empty understanding document on left, chatbot on right
5. The chatbot sends the first message — a question that begins the exploration (driven by Earth 1.0 invisibly)
6. User responds. Chatbot responds. Understanding document grows.
7. This continues as long as the user wants.
8. User can click "Complete Session" when done. Final analysis is generated and saved.

PART 6: DATABASE CHANGES
========================================

6.1 Sessions table changes:
- Add: problem_description (TEXT) — the user's description of their real problem
- Add: thinker_description (TEXT, nullable) — generated on first session completion
- Make puzzle_id nullable (for backward compatibility)

6.2 New table: deep_understanding
- id: UUID (PK)
- session_id: UUID (FK → sessions, cascade delete)
- element: VARCHAR(50) — which element was invisibly applied (earth/fire/air/water/change)
- insight_text: TEXT — the extracted understanding
- created_at: TIMESTAMPTZ

6.3 Modify element_messages table:
- This table currently stores per-element conversations (prompt_index 0-12). 
- The new model has ONE conversation thread, not 13 separate ones.
- Change: prompt_index is no longer needed for the conversation itself, but we still want to track which element was invisibly applied.
- Rename prompt_index to element_applied (VARCHAR(50), nullable) — records which element the backend chose for this exchange
- OR: simply stop using prompt_index and add element_applied as a new column. Keep prompt_index for backward compatibility.

6.4 Remove or deprecate:
- responses table — no longer used (user responses are in element_messages as part of the conversation thread)
- puzzles table — keep for backward compatibility but no longer create new records
- components table — keep, session completion still generates a component
- hints table — no longer used (chatbot responses are in element_messages)

6.5 SQL Migration:

ALTER TABLE sessions ADD COLUMN problem_description TEXT;
ALTER TABLE sessions ADD COLUMN thinker_description TEXT;
ALTER TABLE sessions ALTER COLUMN puzzle_id DROP NOT NULL;

ALTER TABLE element_messages ADD COLUMN element_applied VARCHAR(50);

CREATE TABLE IF NOT EXISTS deep_understanding (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID REFERENCES sessions(id) ON DELETE CASCADE,
  element VARCHAR(50) NOT NULL,
  insight_text TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_deep_understanding_session_id ON deep_understanding(session_id);

PART 7: BACKEND CHANGES
========================================

7.1 NEW: Element Selection Logic

This is the core new piece. When the user sends a message, the backend must decide which element to invisibly apply. 

Create a new function: select_element(problem_description, conversation_history, user_message) -> str

This function makes a Claude call:

---
You are an expert in the 5 Elements of Effective Thinking. Given a user's problem and their conversation so far, determine which element would best serve the user's thinking RIGHT NOW.

The elements are:
- earth: The user needs to ground their understanding. They're missing fundamentals, haven't simplified the problem, or need to add specificity.
- fire: The user needs to try something and fail. They're stuck, overthinking, or afraid to attempt an approach.
- air: The user needs to question their assumptions. They may be solving the wrong problem or missing a fundamental question.
- water: The user needs to see connections and flow. They should map out all approaches, embrace doubt, or follow their best idea further.
- change: The user has gone through significant thinking and needs to reflect on how their understanding has shifted.

The user's problem: {problem_description}
Conversation so far: {conversation_history}
User's latest message: {user_message}

Respond with ONLY the element name in lowercase (earth, fire, air, water, or change). Nothing else.
---

This call is fast (short response) and determines the lens for the chatbot's response.

7.2 REWRITE: Chatbot Response Endpoint

Replace the current GET /session/{id}/analyze endpoint with a new POST /session/{id}/chat endpoint.

Flow:
1. Receive user_message from frontend
2. Save user_message as element_messages row with role="user"
3. Call select_element() to determine which element to apply
4. Build the chatbot prompt using the selected element's definition and coaching guidance
5. Stream the response via SSE
6. Save the completed response as element_messages row with role="assistant" and element_applied={selected element}
7. After response completes, call extract_understanding() in background
8. Save extracted insight to deep_understanding table
9. Return the insight to the frontend so it can update the left column

Chatbot prompt:

---
You are a thinking guide helping someone develop deeper understanding of a problem they're facing. You ask smart, probing questions and make observations that help them see their problem from new angles. You never give them the answer — you help them discover it.

You are invisibly applying the element of {ELEMENT}: {element definition and coaching guidance}

But you NEVER mention this element by name. You never say "let's think about the fundamentals" if that sounds like you're following a script. Instead, naturally ask questions or make observations that embody this element's thinking lens.

The user's problem: {problem_description}
Conversation so far: {full conversation history}
User's latest message: {user_message}

Rules:
1. NEVER mention the 5 Elements, element names, or any framework terminology. The user should not know you're using a framework.
2. Ask ONE focused question or make ONE sharp observation. Not both. Not multiple.
3. Only reference information the user has provided. Never invent details.
4. Keep your response to 40-60 words. Two sentences maximum. Be direct.
5. Sound like a smart colleague thinking alongside them, not a coach or teacher.
6. Do not use markdown formatting.
---

7.3 REWRITE: Session Start

POST /session/start accepts { problem_description }
Creates session with problem_description.
Returns { session_id, problem_description }

After session is created, the backend also generates the FIRST chatbot message — the opening question. This uses Earth 1.0 invisibly: "What do you understand about this problem so far, and where are you stuck?" (but phrased naturally by the LLM, not hardcoded).

7.4 KEEP: Session Complete

POST /session/complete stays. 
On first completed session, also generate thinker_description via a Claude call:

---
Based on this user's conversation about their problem, write a one-sentence professional observation about how they think. Not a gaming archetype. A genuine insight. Examples: "You lead with fundamentals and rarely move forward without solid footing." or "You think by trying — your instinct is to test ideas before analyzing them." Keep it under 15 words.
---

Save to sessions.thinker_description and return it.

7.5 KEEP: Extract Understanding

POST /session/{id}/extract-understanding stays but simplified:
- No longer needs prompt_index
- Takes element (the invisibly selected element) and extracts understanding from the latest exchange

7.6 KEEP: Get Deep Understanding

GET /session/{id}/deep-understanding stays as-is.

7.7 REMOVE: These endpoints

- GET /session/{id}/analyze (replaced by POST /session/{id}/chat)
- POST /session/respond (no longer needed — responses go through /chat)
- GET /prompts (no more visible prompts)
- POST /puzzle/generate
- GET /puzzles
- GET /puzzles/{id}
- GET /puzzles/{id}/solution
- POST /demo/nudge
- GET /session/{id}/nudge-limit (no more nudge limits — it's a conversation now)

7.8 UPDATE: Model

Keep Claude Sonnet 4 (claude-sonnet-4-20250514).
Reduce max_tokens for chatbot responses to 150 (targeting 40-60 words).
Element selection call: max_tokens 10.
Understanding extraction: max_tokens 200.

Keep the full element definitions from Stockfish v3 document (Section 3) with all sub-elements and coaching guidance. These are used by the select_element function and the chatbot prompt. The user never sees them.
