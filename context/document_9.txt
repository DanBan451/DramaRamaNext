MUYAIUM Technical Overview
High-Level Architecture for the AI-Utilization Pivot — Single Phase
1. Overview
This document describes the technical changes required to pivot DramaRama from LeetCode-based puzzles to AI-utilization puzzles (MUYAIUM). This is a single pivot — not a phased rollout. Everything described here is built now.
The existing stack — Supabase, FastAPI, Next.js, Clerk, and Claude API — remains. The changes are concentrated in three areas: puzzle generation, the Teacher Agent, and the nudge system. A background job runner (FastAPI BackgroundTasks) is added for the Teacher Agent.

3. Puzzle Generation (Rewritten)
3.1 Current State
The current system takes a LeetCode algorithm name, sends it to Claude, and receives a story-formatted re-skin. Output contains markdown headers and is plain text only. The user types the algorithm name in the workspace.
3.2 New Approach
Puzzles are no longer generated by the user. The developer generates puzzles via a dev-only button in the app. This button does two things:
Calls Claude to generate an AI-utilization puzzle (scenario + constraints + concrete example)
Triggers the Teacher Agent as a background task to generate 10 flows of ideas for that puzzle
Puzzle structure: The generation prompt instructs Claude to produce a JSON object with three fields: scenario (the real-world AI-utilization situation), constraints (what makes it hard), and example (a concrete, grounded instance). No “hook” field. The puzzle must have at least one concrete solution.
No markdown: The generation prompt must explicitly instruct Claude to NOT use markdown headers, bold text, or any formatting. Plain text only.
No categories or difficulty: Puzzles are simply AI-utilization scenarios. No formal categorization or difficulty levels.
User-facing puzzle selector: The workspace page replaces the text input with a puzzle selector that shows all available pre-generated puzzles. The user picks one and starts a session.

6. Teacher Agent Architecture
The Teacher Agent is built in this pivot. It runs as a background task triggered by the developer’s puzzle generation button.
6.1 Flow Generation
Trigger: Developer clicks the dev-only “Generate Puzzle” button. After the puzzle is created and stored, a FastAPI BackgroundTask is kicked off.
Methodical order: The agent always starts at Earth 1.0 and proceeds in order through all 13 prompts: Earth 1.0 → Earth 2.0 → Earth 3.0 → Fire 1.0 → Fire 2.0 → Fire 3.0 → Air 1.0 → Air 2.0 → Air 3.0 → Water 1.0 → Water 2.0 → Water 3.0 → Change.
Step-by-step with natural continuity: For each prompt, the agent sends Claude the puzzle context, all prior steps and their insights, and the current prompt. Claude responds with the agent’s thinking for that step. Each step’s insight informs the next step. The flow must read as a coherent, connected chain of thinking — not disjointed answers.
Must reach a solution: The agent must arrive at a concrete, definite answer to the puzzle. The solution_reached field stores this. The agent is not mindlessly applying prompts — it is thinking through the puzzle and solving it.
10 flows per puzzle: The background task runs the full 13-step process 10 times. Because the LLM’s responses vary, each flow may surface different insights and arrive at the same or different valid solutions.
Storage: Each flow is stored as a row in teacher_flows with the full step chain (as JSONB) and the solution reached.
6.2 Background Job Runner
Use FastAPI BackgroundTasks. This is the simplest option and works for the current deployment (local uvicorn or a single production server). If the server restarts mid-job, the job is lost — acceptable for now since the developer can re-trigger. No need for SQS, pg_cron, or other infrastructure at this stage.
7. Nudge System Changes
The existing nudge flow stays: user clicks Get Nudge, real-time stream opens, Claude generates a coaching response. The changes are:
Fetch Teacher Agent flows: When the user requests a nudge, the backend fetches all 10 teacher_flows for the puzzle.
Match user responses: Compare the user’s saved responses to each flow’s step sequence. Find the closest matching flow based on keyword overlap and semantic similarity.
Identify next step: From the matched flow, find the next step the user hasn’t reached yet.
Build nudge prompt: Pass the puzzle, the user’s responses, the matched flow’s next step, and the current quality analysis to Claude. Claude generates a coaching response that guides the user toward the next element in the flow without giving away the answer.
Save matched_flow_id: Record which flow the nudge was based on in the hints table.
Nudge limits: 5 nudges per user per puzzle. Unlimited for danieldob200@gmail.com. Enforce in the backend before calling Claude.
Keep the existing quality gate (detect shallow/gibberish responses) and strength analysis as additional context for the nudge prompt.
8. Token Limits
Add token caps to prevent abuse and cost overruns:
Puzzle generation: Max output tokens per puzzle generation call. This is developer-only so less critical, but prevents runaway costs.
Nudge responses: Max output tokens per nudge. This is user-facing — critical to cap.
Teacher Agent flows: Max tokens per step in the 13-step flow. 10 flows × 13 steps = 130 Claude calls per puzzle — set reasonable limits per call.
9. High-Level System Flow
Developer generates a puzzle:
Dev clicks Generate Puzzle button  → Claude creates puzzle (scenario + constraints + example + solution)  → Puzzle stored in puzzles table  → BackgroundTask triggered: Teacher Agent runs 10 flows  → Each flow: 13 steps in methodical order, reaching a solution  → Flows stored in teacher_flows table  → Puzzle now available to users in the puzzle selector
User works through a puzzle:
User selects puzzle from selector → Session created  → Workspace loads (elements sidebar + puzzle + prompts)  → User works through 13 prompts, saving responses  → User clicks Get Nudge (max 5)     → Backend fetches 10 teacher_flows for puzzle     → Matches user responses to closest flow     → Builds nudge prompt with matched flow's next step     → Streams Claude's response to UI in real time  → User arrives at solution → Session saved → Dashboard updates
10. Frontend Changes
Workspace page: Replace the algorithm name text input with a puzzle selector dropdown/list showing all available pre-generated puzzles. Update all labels and copy to reference AI-utilization. Ensure puzzle display has no markdown artifacts.
Prompt text: Update the 13 prompts in entities.py with the new AI-utilization wording. Retain original prompt names.
Elements reference page: Update the /elements page to show the AI-utilization lens for each element.
Nudge limit display: Show remaining nudges (e.g., “3/5 nudges remaining”) on the Get Nudge button.
Dev-only puzzle generator: A button (hidden from regular users, visible to developer account) that triggers puzzle generation + Teacher Agent flows. Could be a route like /dev/generate or a button gated by email check.
Microinteractions: Use Framer Motion for: green flash/animation when a response is saved, progress dot wiggle, confetti overlay on session completion (ConfettiOverlay.jsx already exists), orange/color popups for milestones. These are small dopamine triggers that make the experience addicting.
Typography: Review font sizing using grtcalculator.com for better readability and professional feel.
Landing page and dashboard: Update copy to reference AI-utilization instead of LeetCode/algorithms.
11. Deployment Notes
The backend currently runs locally via uvicorn. For this pivot, the app can remain on its current deployment setup. The Teacher Agent uses FastAPI BackgroundTasks which works with any single-server deployment (local, Railway, Render, VPS). No Lambda, SQS, or additional infrastructure is needed.
If the user base grows significantly, the Teacher Agent job runner can be migrated to a more durable solution. But for the MVP with a small number of puzzles and users, BackgroundTasks is sufficient.
